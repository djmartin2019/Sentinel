# Database & Prisma

Sentinel uses **PostgreSQL** as its database and **Prisma** as the ORM. The Prisma schema, migrations, and client generation all live at the monorepo root so every app in the workspace shares a single source of truth for the data layer.

## Directory layout

```
sentinel/
├── prisma/
│   ├── schema.prisma               # Models, relations, enums
│   └── migrations/
│       ├── migration_lock.toml     # Locks the provider to PostgreSQL
│       └── 20260515000044_init/
│           └── migration.sql       # Initial table definitions
├── generated/                      # Git-ignored — created by pnpm db:generate
└── .env                            # DATABASE_URL lives here (git-ignored)
```

## Schema — `prisma/schema.prisma`

### Generator

```prisma
generator client {
  provider = "prisma-client-js"
}
```

Uses the default Prisma JS client output location (`node_modules/.prisma/client`), which is what `@prisma/client` resolves to. No custom `output` path is set, so the import `from "@prisma/client"` works out of the box.

### Datasource

```prisma
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}
```

The connection string is read from the `DATABASE_URL` environment variable at runtime. In local development this is supplied by the root `.env` file.

## Models

### `MonitoredTarget`

Represents a URL that Sentinel watches.

| Column | Postgres type | Notes |
|---|---|---|
| `id` | `TEXT` (PK) | CUID generated by Prisma |
| `name` | `TEXT` | Human-readable label |
| `url` | `TEXT` | Full URL that gets pinged |
| `interval` | `INTEGER` | Polling interval in seconds (default 30) |
| `createdAt` | `TIMESTAMP` | Set automatically on insert |

Relation: one `MonitoredTarget` → many `HealthCheck` rows.

### `HealthCheck`

One row is written every time a target is checked.

| Column | Postgres type | Notes |
|---|---|---|
| `id` | `TEXT` (PK) | CUID generated by Prisma |
| `targetId` | `TEXT` (FK) | References `MonitoredTarget.id` |
| `status` | `CheckStatus` | `UP` or `DOWN` |
| `statusCode` | `INTEGER` (nullable) | HTTP response code; absent on network errors |
| `latencyMs` | `INTEGER` (nullable) | Round-trip time in ms |
| `errorMessage` | `TEXT` (nullable) | Populated on failures |
| `checkedAt` | `TIMESTAMP` | Set automatically on insert |

### `CheckStatus` enum

```prisma
enum CheckStatus {
    UP
    DOWN
}
```

Maps to a native Postgres `ENUM` type.

## Migrations

Prisma Migrate tracks schema changes as versioned SQL files under `prisma/migrations/`.

### How a migration is applied

1. Prisma reads `schema.prisma` and diffs it against the current database state.
2. A temporary **shadow database** is spun up to validate the SQL before it touches the real database. This requires the database user to have `CREATEDB` privilege.
3. The validated SQL is applied to the real database and the migration is recorded in the `_prisma_migrations` table.

### Current migrations

| Migration | What it does |
|---|---|
| `20260515000044_init` | Creates the `CheckStatus` enum, `MonitoredTarget` table, `HealthCheck` table, and the foreign key constraint |

### Running migrations

```bash
# Apply pending migrations (development — creates migration files automatically)
pnpm prisma migrate dev --name <description>

# Apply pending migrations (production — never creates new migration files)
pnpm prisma migrate deploy
```

## Prisma Client

The generated client is the typed interface the apps use to query the database. It must be regenerated whenever `schema.prisma` changes.

```bash
# From the monorepo root
pnpm db:generate
```

This runs `prisma generate`, which writes the client into `node_modules/.prisma/client`.

### Using the client

The singleton Prisma client is created once in `apps/checker/src/db/prisma.ts` and imported wherever queries are needed:

```ts
import { prisma } from "../db/prisma";

// Fetch all monitored targets
const targets = await prisma.monitoredTarget.findMany();

// Write a health check result
await prisma.healthCheck.create({ data: { ... } });
```

## Local setup

### Prerequisites

- PostgreSQL running locally on port 5432
- A `sentinel` role and database created with the right permissions

```sql
CREATE ROLE sentinel WITH LOGIN PASSWORD 'sentinel' CREATEDB;
CREATE DATABASE sentinel OWNER sentinel;
GRANT ALL PRIVILEGES ON DATABASE sentinel TO sentinel;
GRANT ALL ON SCHEMA public TO sentinel;
```

> `CREATEDB` is required so Prisma Migrate can create the shadow database.

### Environment variable

Add a `.env` file at the **monorepo root**:

```
DATABASE_URL="postgresql://sentinel:sentinel@127.0.0.1:5432/sentinel?schema=public"
```

### First-time initialization

```bash
# 1. Install dependencies
pnpm install

# 2. Generate the Prisma client
pnpm db:generate

# 3. Apply migrations to your local database
pnpm prisma migrate dev
```

After this, the `MonitoredTarget` and `HealthCheck` tables will exist and the checker app can connect.
